# Milestone v0.26.1 — Node-to-Frame Broadcast Fix (patch)

**Status:** in progress

> Opened 2026-07-09 via `/doc-creator-driver:intake` (bug report). Patch release bundling two bugfixes — a regression in `commit_step_frames()`'s live broadcast (B18) and the scheduling of the already-decided-but-unimplemented FR21 (B19). No new features, so it stays a `.1` patch of v0.26 rather than a new minor milestone (user decision: patch releases are for bugfixes until backward compatibility is guaranteed; see `02` §A1). B19 does touch the viewport-cache schema (composite `id:frameIndex` key, user decision) — a data-shape change, not a new architectural direction.

### Sprint 49 — Fix `broadcastStepFrames()` dropping `nodeToFrame` ✅
- [x] **Thread `nodeToFrame` through the live `commit_step_frames()` broadcast.** `server/ws.ts`'s `broadcastStepFrames()` has no `nodeToFrame` parameter and never forwards it into `broadcastReplace()`, so a normal (non-history-reload) commit with `node_to_frame` silently loses the map — clicking a mapped node in the browser does nothing (bug B18 in `01`, root-caused in `03` U4e / `04` §"Unified broadcast projector" / `02` C2e). Add the parameter to `broadcastStepFrames()`'s signature and pass it through from `render-core.ts`'s `commitStepFramesResult()` (which already computes and persists the map correctly — only the final broadcast call is missing it).
  - *Acceptance:* a live `commit_step_frames(id, node_to_frame={...})` broadcasts a `replace` message carrying `nodeToFrame`, identical in shape to what history-reload (`POST /snapshots/load`) already sends for the same content. Clicking a mapped node in a real browser (not just headless/unit) fires `POST /seek` and jumps to the target frame — verified against `tests/human_driven/showcase.js`'s Section 14 (`runNodeToFrameDemo`).
  - *Regression coverage:* new/updated unit test on `broadcastStepFrames()` (or its caller) asserting `nodeToFrame` is present in the emitted broadcast when passed; existing `commit_step_frames` tests updated if they previously asserted (or failed to catch) the omission.
  - **Shipped 2026-07-09:** `broadcastStepFrames()` gained a 6th optional `nodeToFrame` parameter, forwarded straight into `broadcastReplace()`. **Scope turned out wider than the checklist text above:** tracing every call site of the wrapper found the exact same drop in two more places sharing the same root cause — `POST /step` (`server/app.ts`) and the `step()` MCP tool (`server/mcp.ts`) — neither forwarded `state.nodeToFrame` either, meaning a mapped-node click would have stopped working the moment the agent (or Prev/Next) advanced the frame even after the primary commit-broadcast fix. `/seek` and `seek()` were already correct (they call `broadcastReplace()` directly). All three call sites (`commitStepFramesResult`, `POST /step`, `step()`) now pass `nodeToFrame`; `append_frame`'s live-preview broadcast intentionally still doesn't (the map doesn't exist until commit). New unit tests: `tests/unit/server/ws.test.ts` (2 new — forwards/omits), `tests/unit/server/app.test.ts` (2 new — commit broadcast + `/step` advance), `tests/unit/server/mcp.test.ts` (2 new/updated — `step()` advance + commit broadcast). Full suite: 444 passing, `tsc --noEmit` (server + client) and `npm run lint` clean.

> **Implementation note:** consider whether `broadcastStepFrames()`'s parameter list should be typed to statically require every field `broadcastReplace()` accepts (e.g. spreading a shared options object) rather than naming each one — the whole point of the v0.23 unification was to make this class of drift impossible, and today it's still possible to add a field to `broadcastReplace()` without the compiler forcing every wrapper to consider it. Not required for this patch to close, but worth a Design Debt Log entry in `01` if not addressed here.

### Sprint 50 — Per-frame auto-fit for step-frames sequences (FR21, bug B19) ✅
- [x] **Re-fit (or restore a per-frame manual viewport) on every `step()`/`seek()` frame change**, instead of computing the fit once at frame 0 and holding it for the whole sequence. `Mermaid.svelte`'s `isNewSnapshot()`-gated auto-fit check needs to key off `id:frameIndex`, not just `id`, so navigating within an already-seen sequence still triggers a fresh fit (or restore) per frame. Server-side, `POST /viewport` and every broadcast path that reads/writes `viewport-cache.json` (`viewport-cache.ts`) switch from a bare `id` key to composite `id:frameIndex`; `POST /viewport`'s body gains a `frame` field. The two delete endpoints (`POST /snapshots/delete-files`, `POST /snapshots/delete-workspace`) switch their cache cleanup from an exact-key match to a `id:`-prefix match, since one snapshot id can now own multiple per-frame cache entries. See `03` F19, `04` "Mermaid Viewport Persistence", `02` C3.
  - *Acceptance:* a step-frames sequence with frames of very different intrinsic size (e.g. a tall sequence diagram followed by a wide flowchart) fits correctly on every frame during `step()`/`seek()` navigation, not just frame 0. Manually zooming/panning on one frame and navigating away then back to that exact frame (same session or after a history reload) restores that frame's own manual viewport, independent of other frames in the sequence. Verified live in a real browser (not just headless), per `02` C3's note that this class of bug previously had an automated-testing blind spot.
  - *Regression coverage:* unit tests for `viewport-cache.ts` covering the composite key read/write and prefix-match delete cleanup; e2e test navigating a multi-frame sequence with differently-sized frames and asserting per-frame fit.
  - **Shipped 2026-07-09:** `viewport-cache.ts`'s `getViewport`/`setViewport` now take `(id, frameIndex)`; `deleteViewports` does a prefix match on `"<id>:"` (a dedicated test guards against an id being a prefix of a different id's key, e.g. `"id-1"` vs `"id-10:0"`). `broadcastStepFrames()` (`server/ws.ts`) gained a 7th optional `viewport` parameter. Server-side, every frame-navigation path now looks up `getViewport(id, frameIndex)` and forwards the result: `POST /step`, `POST /seek`, `step()`/`seek()` MCP tools (all previously never looked up a viewport at all — only history-reload did), and `POST /snapshots/load` updated to the composite key (always frame 0, since history-reload always redisplays from frame 0). `POST /viewport`'s body gains a required `frame` field, validated as a non-negative integer. Client-side: `Mermaid.svelte` gained a `currentFrame` prop (threaded through `App.svelte` → `registry.ts`'s `RendererContext` → the mermaid entry's `props()`); `isNewSnapshot()` now compares a composite `"<id>:<frame>"` key (`lastSnapshotKey`) instead of `id` alone, so navigating to a different frame of an already-seen sequence now triggers a fresh fit/restore instead of being treated as "continuation, leave the transform alone"; `reportViewport()`'s POST body includes `frame: currentFrame`. Rewrote the e2e test that previously asserted the *old* behavior ("step()/seek() within a sequence preserves the live zoom/pan") into one asserting the new per-frame independence: zooming frame 0, navigating to a visibly bigger frame 2 gets its own fresh auto-fit (not frame 0's zoom), and navigating back to frame 0 restores frame 0's own manual viewport. New/updated unit tests: `viewport-cache.test.ts` (composite key round-trip, per-frame independence, prefix-match delete, prefix-collision guard), `ws.test.ts` (viewport forwarding), `app.test.ts` + `mcp.test.ts` (per-frame viewport lookup on `/step`/`/seek`/`step()`/`seek()`, `POST /viewport`'s `frame` validation), `Mermaid.test.ts` (composite-key restore trigger, `frame` in the `/viewport` POST body), `registry.test.ts` (`currentFrame` prop threading). Full suite: 459 unit tests passing (up from 444), e2e 37/38 (the one failure is the pre-existing dev-server-startup race documented in prior sprints, confirmed passing in isolation and unrelated to this change), `tsc --noEmit` (server + client) and `npm run lint` clean.

- `broadcastStepFrames()` forwards `nodeToFrame` on every call that provides one; live `commit_step_frames()` broadcasts and history-reload broadcasts are at parity for this field.
- `runNodeToFrameDemo` (showcase Section 14) verified working end-to-end against a live browser: clicking each mapped node (Client/API Gateway/Database) jumps to its frame.
- Step-frames sequences re-fit (or restore a manual viewport) per frame during `step()`/`seek()`, not just once per sequence; viewport-cache uses the composite `id:frameIndex` key end-to-end (write, read, and delete-cleanup paths).
- Full unit + e2e suite green.
- `03`/`04`/`02` entries for B18/U4e/C2e and B19/F19/FR21/C3 updated from "known gap"/"scheduled" to resolved.
