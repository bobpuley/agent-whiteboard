## 1. MCP Tool Surface

The MCP server exposes tools to the agent.

| Tool | Signature | Description | Phase |
|------|-----------|-------------|-------|
| `render` | `render(type, payload, options)` | Push content to the canvas. `type` selects the renderer (e.g. `"mermaid"`). Always replaces the current canvas state in v1. `options.title` (optional string) displays a label above the canvas. `options.theme` and action-variant options deferred to planned. **`render()` is single-frame only (v0.26 Sprint 45):** `type: "step-frames"` no longer exists as a top-level content type — a multi-frame Presentation is created exclusively via `init_step_frames`/`append_frame`/`commit_step_frames` (see below); the one-shot `render(type="step-frames", ...)` path is removed, no back-compat shim (`02` N4). `options.node_to_frame` moved to `commit_step_frames` (see below) since it no longer has a `render()` entry point. `options.workspace` **(required, v0.7)** — workspace name for snapshot routing; no fallback; missing value returns `{ ok: false, error: "workspace is required" }`. **v0.11:** success response includes `{ ok: true, id: "<uuid>" }` — the UUID of the snapshot written for this render call. | v0.1; workspace required v0.7; id in response v0.11; single-frame only v0.26 |
| `clear` | `clear()` | Reset the current session canvas. | v0.1 |
| `export` | `export([id])` | Return a canvas source spec. Without `id`: returns the verbatim last `render()` payload (current behavior). With optional `id` (UUID): looks up the snapshot with that ID and returns its payload — the snapshot must have an `id` field (written from v0.11 onward). Response: `{ "ok": true, "data": "<source>" }`. Empty string if canvas is blank (no-id case). Error if id provided but not found. Binary export is planned. | v0.1; id param v0.11 |
| `step` | `step(direction)` | Advance (`"next"`) or rewind (`"prev"`) a step-through sequence. | v0.1 |
| `slideshow` | `slideshow(slides, delay_ms)` | Load a playlist of slides (`[{ type, payload, title? }]`) and auto-advance the canvas on a server-side timer at `delay_ms` intervals. A new call cancels any running slideshow. | v0.2 |
| `slideshow_stop` | `slideshow_stop()` | Cancel the running slideshow timer; last rendered slide remains on screen. | v0.2 |
| `wait_done` | `wait_done()` | Block until the user clicks the Done button in the browser. Returns `{ "ok": true }` when the user signals they are ready to continue. Times out after 10 minutes (returns `{ "ok": true }` regardless). **Arming `wait_done()` supersedes a pending `wait_click()` (v0.26 Sprint 47, OQ11)** — see `wait_click` below; `wait_done()`'s own return shape is unaffected. Intended usage: `render(...)` → `wait_done()` → continue lesson. | v0.2 ✅; supersedes pending wait_click v0.26 |
| `seek` | `seek(frame)` | Jump the step-frame cursor to an arbitrary frame index without repeated `step()` calls. Returns `{ "ok": true, "current_frame": N, "total_frames": M }`. Error if no `step-frames` sequence is loaded or frame is out of range. | v0.2 |
| `wait_click` | `wait_click([node_actions])` | Arm the browser for a single node or edge click on the current Mermaid diagram (plain or step-frames). Applies to `graph`/`flowchart` diagrams; other Mermaid types are best-effort. Optional `node_actions` (Sprint 14): map of node ID → string array; browser shows a popup menu for nodes with registered actions; user selects one. Returns `{ "ok": true, "type": "node"\|"edge", "id": "<id>", "label": "<label>", "action": "<chosen action or null>" }` (action field always present; null when no menu was shown or user clicked without selecting; string value when `node_actions` was provided and user selected an item). On timeout after 10 minutes: `{ "ok": true, "type": "timeout" }`. **Supersession (v0.26 Sprint 47, OQ11):** only one `wait_click()` may be pending at a time — a second `wait_click()` call, or an arming `wait_done()` call, now supersedes it: `{ "ok": true, "type": "superseded" }`, distinct from the plain-inactivity `"timeout"`. Usage: `render(...)` → `wait_click(node_actions?)` → agent handles click result. | v0.2; supersession v0.26 |
| `init_step_frames` | `init_step_frames(frame_type, workspace, title?)` | Begin an incremental step-frames sequence. Creates an empty skeleton in server memory, pushes a placeholder render to the browser (0-frame indicator with optional title), and returns a unique ID. `frame_type` is the content type shared by all frames (e.g. `"mermaid"`). `workspace` is required (same rules as `render()`). `title` is optional. Returns `{ "ok": true, "id": "<uuid>" }`. Error if `workspace` is absent/invalid or `frame_type` is unsupported. The ID expires after 30 minutes of inactivity. | v0.8 |
| `append_frame` | `append_frame(id, payload, label?, type?)` | Append one frame to an in-progress step-frames sequence identified by `id`. `payload` is validated against `type ?? frame_type` using the same per-type hard gate `render()` applies to a single mermaid/vega-lite/etc. payload — **same `validateFrame()` call every content path uses (U2, v0.26 Sprint 39/45); the one-shot `render(type="step-frames")` path this used to be compared against no longer exists.** `label` is an optional display caption for this frame. `type` (v0.17, optional) overrides the sequence's `frame_type` for this one frame — lets a sequence mix content types. Returns `{ "ok": true, "frame_count": N }` where N is the current total number of frames. **After each valid append, immediately pushes the full accumulated partial step-frames sequence to the browser, positioned at the latest frame index N-1** — the user sees the sequence grow one frame at a time. Invalid payloads are rejected before any broadcast; prior frames and browser state are preserved. Error if the `id` is unknown/expired or the payload fails validation. | v0.8; live preview v0.9; per-frame `type` v0.17 |
| `commit_step_frames` | `commit_step_frames(id, node_to_frame?)` | Finalise an in-progress step-frames sequence. **The primary visual has already been rendered incrementally by `append_frame()`** — `commit_step_frames()` is responsible for finalization only: assembles the full step-frames JSON, writes a snapshot, updates in-memory canvas state (so `export()` returns the complete assembled JSON), cancels any running slideshow, and deletes the builder entry. Still pushes a final WebSocket broadcast as part of the standard pipeline (for consistency and to handle edge cases such as `clear()` between appends). `node_to_frame` (optional, v0.26 Sprint 45 — moved here from the now-removed `render(type="step-frames", options.node_to_frame)` path): declarative node ID → frame index map; browser navigates frames on click autonomously (see U4e). Returns `{ "ok": true }`. Error if the `id` is unknown/expired or the sequence has zero frames. After commit, `export()` returns the assembled full step-frames JSON. `clear()` during an active session does NOT cancel the builder entry — TTL handles cleanup. | v0.8; finalization-only v0.9; `node_to_frame` param v0.26 |
| `list_snapshots` | `list_snapshots(workspace)` | **(v0.15)** List the snapshots stored for a given workspace, for the agent to build an `export_html()` request. `workspace` is required (same safety check as `render()` — F14). Returns `{ "ok": true, "snapshots": [{ "id", "timestamp", "type", "title"? }] }`, sorted newest-first (same ordering as `GET /snapshots`). Returns an empty array if the workspace has no snapshots or does not exist. Every snapshot has an `id` since v0.11 (see J1, L5 in `02`); pre-v0.11 files were backfilled. | v0.15 |
| `export_html` | `export_html(workspace, ids, output_path?)` | **(v0.15)** Agent-facing equivalent of the browser's "Export selected" (F17): assembles the same self-contained HTML export for 1..N snapshots in `workspace`, identified by `id` (from `list_snapshots()`). `workspace` is required (same safety check as `render()`). `ids` is a non-empty array of snapshot UUIDs, all scoped to `workspace`; unknown/invalid ids are skipped (same as unreadable-item handling in `POST /export-html`) — if zero valid ids remain, returns `{ ok: false, error: "no valid items to export" }`. Rather than returning the HTML inline (which can be several MB once the `mermaid.js` bundle is embedded — see L1/L6 in `02`), the server writes the assembled HTML to disk and returns `{ "ok": true, "path": "<absolute path>" }`. `output_path` (optional string): if provided, the server creates parent directories as needed (`mkdir -p` semantics) and writes there — no path restriction (see L6 in `02`); **relative paths resolve against the server process's working directory, not the agent's** — pass an absolute path for a specific location. If omitted, defaults to `<WHITEBOARD_SNAPSHOTS_DIR>/<workspace>/exports/<name>-YYYYMMDD-HHmmss.html` (same naming convention as the browser download filename). | v0.15 |

## 3. Transport

| Layer | Mechanism | Role |
|-------|-----------|------|
| Agent → Server | MCP (primary) | Agent calls tools; server executes render commands |
| Server → Browser | WebSocket (`/stream`) | Incremental, real-time updates pushed to the SPA |
| Agent → Server (alt) | REST `POST /render`, `POST /clear`, `GET /export`, `POST /step`, `POST /seek`, `POST /slideshow`, `POST /slideshow/stop`, `POST /wait-done`, `POST /wait-click`, `POST /step-frames/init`, `POST /step-frames/:id/frame`, `POST /step-frames/:id/commit` | Low-level fallback; also usable via `curl` for debugging. `POST /slideshow` and `POST /slideshow/stop` added in v0.2 alongside the slideshow MCP tools. `POST /step` body: `{ "direction": "next" \| "prev" }`. `POST /seek` body: `{ "frame": N }`; jumps step-frame cursor, returns `{ ok: true, current_frame: N, total_frames: M }` or error. `POST /wait-done` long-polls until the user clicks Done. `POST /wait-click` long-polls until a node/edge click; broadcasts `set_node_actions enabled:true` before suspending and `enabled:false` after resolution. `POST /step-frames/init` body: `{ frame_type, workspace, title? }` → `{ ok, id }` (v0.8). `POST /step-frames/:id/frame` body: `{ payload, label? }` → `{ ok, frame_count }` (v0.8). `POST /step-frames/:id/commit` no body → `{ ok }` (v0.8). `GET /snapshots?workspace=<name>` — explicit workspace query param for agent use (v0.15, F11). `POST /export-html` accepts `{ workspace, id }` items in addition to `{ workspace, filename }` (v0.15, F17). |
| Browser → Server | `POST /user-done` | Browser Done button fires this; server calls `signalDone()` to wake any pending `wait_done()` calls, then optionally forwards to the channel relay. |
| Browser → Server | `POST /node-click` | Browser fires when user clicks a node or edge (while `wait_click()` is armed). Body: `{ type: "node"\|"edge", id, label, action? }`. Server calls `signalClick(event)` to resolve pending `wait_click()` calls. Returns `{ ok: true }`. |
| Browser → Server | `POST /snapshots/delete-files` | Delete one or more snapshot files. Body: `{ workspace: "...", filenames: ["...", ...] }`. Validates workspace (safe-name check, must exist under `WHITEBOARD_SNAPSHOTS_DIR`) and each filename (must match `*_screen.json`, no path traversal). Deletes matching files; missing files are silently skipped. Returns `{ ok: true, deleted: N }` where N is the count of files actually removed. Handles single delete (1 filename) and multi-select delete (multiple filenames) identically. (v0.12) |
| Browser → Server | `POST /snapshots/delete-workspace` | Delete a workspace directory and all its contents. Body: `{ workspace: "..." }`. Validates workspace via `isValidWorkspaceName()` (safe-name check, must exist) and additionally asserts the resolved path stays strictly inside the snapshots root before deleting (hardened v0.18, closes B6 — a bare `"."` previously bypassed the weaker ad hoc check in place and resolved to the snapshots root itself). Removes the entire workspace directory recursively. Returns `{ ok: true }`. If the deleted workspace matches `lastWorkspace`, `lastWorkspace` is reset to `""`. (v0.12; workspace-check hardened v0.18) |
| Browser → Server | `POST /export-html` | Export selected snapshots to a self-contained HTML file. Body: `{ items: [{ workspace: "...", filename: "..." }] }`. Validates workspace and filename safety (same rules as `POST /snapshots/load`). Unreadable or malformed snapshot files are silently skipped. If no valid items remain after skipping: `{ ok: false, error: "no valid items to export" }` (400). On success: `Content-Type: text/html; charset=utf-8`, `Content-Disposition: attachment; filename="<name>-YYYYMMDD-HHmmss.html"`. Single-workspace exports use the workspace name (sanitized: non-alphanumeric/`-`/`_`/`.` chars replaced with `-`, truncated to 24 chars); multi-workspace exports use the literal string `export`. Rendering is server-side and static for KaTeX/Vega-Lite/SVG/HTML; Mermaid is embedded client-side and rendered on open (see F17, fixed v0.14). (v0.13; Mermaid rendering fixed v0.14) |
| Server → Browser (WebSocket) | `{ action: "set_node_actions", node_actions, enabled }` | Sent when `wait_click()` is called. `node_actions`: map of node ID → string array (empty map = any click accepted, no popup). `enabled: true` arms the click listener; `enabled: false` disarms it (sent after click resolves or on timeout). |
| Server → Browser (WebSocket) | `{ action: "set_done_armed", armed: boolean }` | Sent when `wait_done()` is called (`armed: true`) and when it resolves or times out (`armed: false`). Also sent to every new WebSocket connection with the current state (see H8). Browser shows the Done button only when `armed` is true. (v0.12) |
| Browser → Server | `POST /viewport` (v0.19) | Reports the current Mermaid zoom/pan state for the currently-displayed snapshot, debounced ~800ms after the last zoom/pan input. Body: `{ id: "<uuid>", scale: N, positionX: N, positionY: N }` (`positionX`/`positionY` normalized fractions of the canvas container). Server writes/updates the entry for `id` in the viewport-cache file. Returns `{ ok: true }`. See F19 (`03`), C3 (`02`). |
| Server → Browser (WebSocket) | `viewport` field on `{ action: "replace", ... }` (v0.19) | When broadcasting a `render()`/`commit_step_frames()` result or a `POST /snapshots/load` reload, the server includes an optional `viewport: { scale, positionX, positionY }` field if a cache entry exists for that snapshot's `id`. Browser applies it instead of auto-fitting. Absent when no cache entry exists (browser auto-fits instead). |

File-system watch (`CLAUDE_SCREEN.md`) is **dropped** — superseded by MCP.

## 4. Session Model

- Sessions are **in-memory**, scoped to a single focused explanation.
- `clear()` resets the canvas; server restart clears everything.
- No cross-session persistence in v1 — deferred to planned.
- The agent is stateless with respect to the whiteboard: it sends commands forward-only and keeps its own record in the terminal (Mermaid source, JSON spec printed alongside the render).

## 5. Functional Requirements

### Communication & Integration

| ID | Requirement | Phase |
|----|-------------|-------|
| F1 | Accept content via MCP tool calls (primary) and REST fallback endpoints (`POST /render`, `POST /clear`, `GET /export`, `POST /step`) | v0.1 |
| F2 | Support rendering types: Mermaid, SVG, HTML, KaTeX, Vega-Lite. D2 deferred (requires server-side render process). | v0.1 |
| F3 | Full-spec replace: agent always sends the complete updated spec; per-element mutation deferred to planned | v0.1 |
| F3a | Validation is a hard gate: invalid payloads are rejected and returned as `{ ok: false, error: "..." }` to the agent; nothing is pushed to the browser and canvas state is unchanged | v0.1 |
| F3a-gap | **Resolved (v0.17, was B5, found 2026-07-03):** the hard gate in F3a is now fully honored for `step-frames` frame content on both creation paths. `validatePayload()`'s `step-frames` branch validates every frame's payload against its effective type (`frame.type ?? frame_type`) — a syntactically invalid mermaid or vega-lite frame is rejected before anything is accepted, on `render(type="step-frames")`, `append_frame()`, `POST /slideshow`, and `POST /snapshots/load` alike (all route through the same shared `validatePayload()`). Previously only `append_frame()` validated per frame. | resolved v0.17 |
| F4 | REST endpoints are `curl`-friendly fallbacks for agents that do not support MCP and for manual debugging. They mirror the MCP tool surface for core operations (`POST /render`, `POST /clear`, `GET /export`, `POST /step`) but do not expose every MCP feature. Primary path is MCP → WebSocket `/stream`. `POST /wait-click` accepts an optional `node_actions` body (`Record<string, string[]>`); the server broadcasts it to the browser via `set_node_actions` so the popup menu appears for registered nodes — same behaviour as the MCP `wait_click(node_actions)` tool. Invalid `node_actions` returns `{ ok: false, error: "..." }` with 400. | v0.1 |
| F5 | Session management with cross-session persistence (`session_id`, history across restarts) | planned |
| F7 | Slideshow: `POST /slideshow` (and `slideshow()` MCP tool) accepts `{ slides: [{ type, payload, title? }], delay_ms, workspace }`. **`workspace` is required (v0.22, FR20 in `01`)** — same validation rule as `render()` (F14); previously `slideshow()` had no workspace parameter at all. Validation (same hard gate as `POST /render`): each slide's `type` and `payload` are validated; `title` (if present) must be a string; if *any* slide fails validation, the entire slideshow request is rejected with `{ ok: false, error: "..." }` and no timer starts. On success, a server-side timer auto-advances the canvas. Each slide is broadcast to the browser using the **same WebSocket event format** that `POST /render` would produce for that slide's type — **including the `id` field (fixed v0.22, was B15 in `01`)**: every slide broadcast carries a fresh snapshot id, matching `commitRenderResult()`'s contract so F19's auto-fit/restore logic applies uniformly regardless of whether the content came from `render()` or a slideshow tick. **A slide is always exactly one frame (v0.26 Sprint 45):** `type: "step-frames"` no longer exists as a top-level content type, so a slide can no longer itself expand into per-frame ticks — the earlier "step-frames slide expands into N timer ticks" behavior is removed along with it; each slide is one tick, one to one. Manual Prev/Next navigation remains functional during and after the slideshow (against whatever step-frames sequence is separately loaded via `commit_step_frames()`, if any). A new call cancels any running slideshow; `POST /render` and `POST /clear` also cancel it. At most one active slideshow at a time. **Persistence (v0.22, FR20 in `01`):** individual ticks are never written to disk — when the slideshow session ends (runs to completion, is stopped via `slideshow_stop()`, or is superseded by a new `render()`/`slideshow()` call), whatever is last on screen is persisted as **one** snapshot via `saveSnapshot()`, reusing the same `id` already broadcast live. Mirrors `commit_step_frames()`'s "transient until finalized" pattern (F15). `clear()` ends a running slideshow **without** persisting (F10's "clear() never produces a snapshot" still holds). ✅ Implemented in Sprint 9 (2026-05-31); `id` parity, required `workspace`, and finalize-on-end persistence all added v0.22; step-frames-slide expansion removed v0.26. | v0.2; workspace required + persistence v0.22; step-frames-slide expansion removed v0.26 |
| F8 | Slideshow stop: `POST /slideshow/stop` (and `slideshow_stop()` MCP tool) cancels the running timer; last rendered slide remains on screen. No-op if no slideshow is running. Note: `POST /render` and `POST /clear` also cancel any running slideshow (canvas ownership transfers to agent). `POST /step` and `POST /seek` do not cancel slideshow. | v0.2 |
| F9 | Done signal: `POST /user-done` (browser button) wakes all pending `wait_done()` MCP tool calls via an in-process EventEmitter. `POST /wait-done` (REST) long-polls until the signal fires or the 10-minute timeout elapses. Multiple concurrent `wait_done()` calls are all resolved simultaneously by a single click. | v0.2 ✅ |
| F6 | HTML/SVG payloads must be sanitized with DOMPurify in the browser before render; sanitization is silent (cleaned output rendered, no error state). No server-side hard gate for HTML/SVG — the `type` field is validated but the payload is passed through. | v0.1 |
| F10 | **Render snapshot persistence:** after every successful `render()` call (i.e. payload passes validation), the server writes a JSON snapshot file to `<snapshots_dir>/<workspace>/<timestamp>_<id>_screen.json`. **Snapshot schema (v0.26 Sprint 43 — unified `frames[]` schema, replacing the old top-level `type`/`payload`/`options` triple):** `{ "id": "<uuid>", "timestamp": "<ISO 8601>", "workspace": "<name>", "cursor": 0, "frames": [{ "type", "payload", "label"? }, …], "title"?: "<string>", "nodeToFrame"?: { … }, "rawPayload"?: "<verbatim step-frames envelope JSON>" }`. A one-shot render is a single-element `frames` array; a step-frames sequence is the full multi-element array (each frame's already-resolved effective type — `frame.type ?? frame_type`), `cursor` is always `0` at write time (history load always redisplays from the first frame). `rawPayload` is present only when `frames.length > 1`, kept solely so `export(id)` can return byte-identical content instead of a reconstruction; a committed 1-frame step-frames sequence collapses into a plain single-frame record with no `rawPayload` (same "indistinguishable from a one-shot render" policy as the WS contract, Sprint 42). A one-time migration script (`server/migrate-snapshots.ts`) upgrades pre-Sprint-43 files to this schema — no dual-read path exists for the old shape (OQ5a in `02`). `step()`, `seek()`, `clear()`, and failed `render()` calls do not produce snapshot files. **`slideshow()` (v0.22, F7, FR20 in `01`):** individual ticks are likewise not snapshotted; exactly one snapshot is written when the slideshow session ends, capturing whatever was last on screen — see F7. Snapshot directory root defaults to `~/.agent-whiteboard/`; overridable via `WHITEBOARD_SNAPSHOTS_DIR` env var (for testing and custom setups). Workspace is always supplied by the agent via `options.workspace` (mandatory since v0.7); no env var or implicit derivation. Directory is created if it does not exist (`mkdir -p` semantics). A write failure must never block rendering — the server logs a warning to stderr and continues. No read/resume API in v1 of this feature (write-only). | v0.3; unified `frames[]` schema v0.26 |
| F11 | **Snapshot list endpoint:** `GET /snapshots` — returns the list of snapshot files for a workspace, sorted by timestamp descending. Response: `{ ok: true, snapshots: [{ id, filename, timestamp, type, title? }] }`. `title` is included only if the snapshot file's top-level `title` field was present and non-empty (v0.26 Sprint 43 — was `options.title` before the unified `frames[]` schema). `type` is a derived display badge: `"step-frames"` for a multi-frame sequence, else the single frame's own resolved type — the schema has no top-level `type` field anymore. Returns an empty array if no snapshots exist or the snapshot directory is absent. Snapshot directory root read from `WHITEBOARD_SNAPSHOTS_DIR` env var (or `~/.agent-whiteboard/` default). Workspace resolution: an explicit `?workspace=<name>` query param, if provided, is used (validated with the same safety check as F12); otherwise defaults to `lastWorkspace` — the workspace from the most recent successful `render()` call in the session (G2c). Unreadable or malformed snapshot files are skipped with a warning to stderr. **v0.15:** each entry gains an `id` field (the snapshot's UUID, see J1) and the `?workspace=` param — both purely additive; the browser's existing call pattern (no query param, filename-keyed UI) is unaffected. `list_snapshots(workspace)` (MCP tool, v0.15) wraps this endpoint with a mandatory `workspace` argument for agent use — see `list_snapshots` in the MCP Tool Surface table. | v0.4; `id` + explicit `?workspace=` v0.15 |
| F12 | **Snapshot load endpoint:** `POST /snapshots/load` — body: `{ "filename": "…" }` (current workspace) or `{ "filename": "…", "workspace": "…" }` (explicit workspace). Filename safety check: must match `*_screen.json`, no `/` or `..`. ## 5b. Tool Error Scenarios

**Agent error recovery:** The agent is responsible for deciding how to handle errors returned by the server. The server returns `{ ok: false, error: "<message>" }` for validation failures and certain invalid states; the agent can inspect the error and retry, recover, or fail gracefully.

| Tool | Success response | Error cases |
|------|------------------|------------|
| `render()` | `{ ok: true }` | Invalid payload (keyword, syntax, JSON, format) → `{ ok: false, error: "..." }`. Unsupported/multi-frame `type` (e.g. `"step-frames"`, removed v0.26) → `{ ok: false, error: "type must be one of: ..." }` |
| `clear()` | `{ ok: true }` | Never fails (always succeeds) |
| `export()` | `{ ok: true, data: "<source>" }` | Never fails; returns empty string if canvas is blank |
| `step()` | `{ ok: true, current_frame: N, total_frames: M }` | No step-frames loaded → `{ ok: false, error: "..." }`. Direction invalid → `{ ok: false, error: "..." }` |
| `seek()` | `{ ok: true, current_frame: N, total_frames: M }` | No step-frames loaded → `{ ok: false, error: "..." }`. Frame out of range → `{ ok: false, error: "..." }` |
| `wait_done()` | `{ ok: true }` | Never fails; times out after 10 minutes (returns `{ ok: true }` regardless) |
| `wait_click()` | `{ ok: true, type, id, label, action }` (`action` always present: null when no popup shown, string when menu item selected) or `{ ok: true, type: "timeout" }` or `{ ok: true, type: "superseded" }` | Never fails; times out after 10 minutes, returning `{ ok: true, type: "timeout" }`; superseded by a new `wait_click()` or an arming `wait_done()` (v0.26 Sprint 47), returning `{ ok: true, type: "superseded" }` |
| `slideshow()` | `{ ok: true }` | Any slide fails validation → `{ ok: false, error: "..." }` (entire slideshow rejected, no timer started). Missing/invalid `workspace` → `{ ok: false, error: "..." }` (v0.22). |
| `slideshow_stop()` | `{ ok: true }` | Never fails; no-op if no slideshow running |
| `init_step_frames()` | `{ ok: true, id: "<uuid>" }` | Unsupported `frame_type` → `{ ok: false, error: "..." }`. Missing/invalid `workspace` → `{ ok: false, error: "..." }`. |
| `append_frame()` | `{ ok: true, frame_count: N }` | Unknown/expired ID → `{ ok: false, error: "step-frames session not found or expired" }`. Invalid payload → `{ ok: false, error: "..." }`. |
| `commit_step_frames()` | `{ ok: true }` | Unknown/expired ID → `{ ok: false, error: "step-frames session not found or expired" }`. Zero frames → `{ ok: false, error: "cannot commit empty step-frames sequence" }`. Invalid `node_to_frame` shape (REST only — not a map of string → number) → `{ ok: false, error: "..." }` (v0.26). |
| `list_snapshots()` | `{ ok: true, snapshots: [...] }` (empty array if none) | Missing/invalid `workspace` → `{ ok: false, error: "..." }` |
| `export_html()` | `{ ok: true, path: "<absolute path>" }` | Missing/invalid `workspace` → `{ ok: false, error: "..." }`. Empty `ids` array → `{ ok: false, error: "..." }`. All `ids` unresolvable in `workspace` → `{ ok: false, error: "no valid items to export" }`. Disk write failure (e.g. permission denied at `output_path`) → `{ ok: false, error: "..." }`. |

**Error recovery strategies:**
- Invalid render payload: agent can inspect the error and either fix the syntax or escalate to the user
- Step out of range: agent should clamp to `[0, totalFrames-1]` or inform the user
- No step-frames loaded: agent attempted to step/seek without first loading a step-frames sequence; check render history or render a sequence before step/seek
- Slideshow validation fails: check the error message for which slide failed; agent can fix that slide and retry the entire slideshow

## 5c. Known Gaps (from code review intake, 2026-07-04) — resolved v0.18

> Found during a Node.js/TS + frontend code review pass (2026-07-04); logged as B6–B14 in `01_input-ideas.md`. All nine fixed in Milestone v0.18 (`05_dev-plan.md`, Sprint 31). These were gaps against existing (correct) requirements, not new scope; the table below is kept as a record of what was gapped and how each was closed.

| Bug | Requirement touched | Gap | Resolution (v0.18) |
|-----|---------------------|-----|---------------------|
| B6 | §3 Transport table, `POST /snapshots/delete-workspace` ("Validates workspace (safe-name check, must exist)") | The endpoint's actual workspace check is a weaker ad hoc inline check, not the correct `isValidWorkspaceName()` — it doesn't reject a bare `"."`, which resolves to the snapshots root and gets recursively deleted. | `GET /snapshots`, `POST /snapshots/load`, `POST /export-html`, and both delete endpoints now all route through `isValidWorkspaceName()`; `POST /snapshots/delete-workspace` additionally asserts the resolved path stays strictly inside the snapshots root before `rmSync`. |
| B7 | F10 (Render snapshot persistence) | Filename is timestamp-only (second precision) with no uniqueness guarantee; two writes in the same second collide and silently overwrite. | Filename now includes the snapshot's own `id` UUID alongside the timestamp — two same-second writes always produce distinct files. |
| B8 | V1 / V5 (Diagram rendering, step-through frames) | No requirement states that an in-flight async render must be discarded if a newer one starts before it resolves; renderers can display stale content with no ordering guard. | `Mermaid.svelte` and `VegaLite.svelte` (the two renderers with a genuine async gap between starting a render and touching the DOM/view) now discard a render's result if a newer render has since started. |
| B9 | U4a (Done button) | No requirement covers failure handling for the `POST /user-done` call; a rejected fetch has no recovery path today. | `handleDone()` catches a failed request, shows a "Failed ✗" state for 2s, and leaves the button enabled for retry instead of an unhandled rejection. |
| B10 | Implied by `04`'s `strict: true` TypeScript config (documented for server only) | The strict-typing guarantee was never extended to the client; nothing type-checks `client/` in the build today. | `svelte-check` runs against `client/tsconfig.json` via `npm run typecheck`, chained into `npm run build` — a client type error now fails the build exactly as a server one does. |
| B11 | §3 Transport, WebSocket message contract | No requirement specifies handling for an unrecognized/malformed `type` in a WS message; behavior today is a silent no-op. | `ws.ts` validates the message shape before dispatch; an unrecognized action/type is logged via `console.error` and dropped instead of silently propagating. |
| B12 | U7 / U7i (History panel, delete/export modal) | No requirement specifies keyboard-accessibility behavior (focus trap, Escape) for dialogs, including the destructive delete flow. | Both dialogs gained `aria-modal="true"` and a shared `trapFocus` Svelte action: Escape closes, Tab/Shift+Tab cycles within the dialog, focus moves in on open and restores on close. |
| B13 | U7 (History panel) | No requirement specifies that a failed `GET /snapshots/all` must be surfaced to the user at every call site; `App.svelte`'s copy currently fails silently. | Both call sites now share `fetchAllSnapshots()`; a failure is visibly surfaced in both the history panel and the delete/export modal (new `loadError` prop). |
| B14 | F17 (`POST /export-html`) | No requirement addresses concurrent export calls; `generateExportHtml()`'s global DOM patching is not documented as, or made, reentrant-safe. | `generateExportHtml()` calls are now serialized via a simple promise queue, so only one call's global DOM patching is ever active at a time. |

## 6. Out of Scope for v1

- Multiple named panels/tabs: planned (one canvas at a time in v1)
- Terminal ASCII fallback: planned (browser always assumed available in v1)
- Node/edge click interactions (`wait_click()`): shipped in v0.2 (Sprints 12–14). Basic "user is done" signal (`wait_done()` + Done button) shipped in Sprint 10.
- Slider/quiz widgets → agent: planned (after node clicks).
- Cross-session persistence / history across restarts: write-only snapshot persistence shipped in v0.3 (F10); user-facing history browser (read/navigate) shipping in v0.4 (F11–F12, U7). Agent-facing history query (MCP tool to list or reload past sessions) remains planned.
- Binary export (PNG/SVG/PDF): planned
- D2 renderer: planned (requires server-side render process)
- Concurrent browser connections / multi-tab state sync: planned (second tab starts blank in v1)
- WebSocket reconnection state replay: planned — on disconnect the browser clears the canvas and displays "Server disconnected. Restart `npm run dev`." No auto-retry in v1.
- Free-form canvas / whiteboard drawing: deferred (nice-to-have)
- Agent error-recovery behavior: out of scope — the server returns `{ ok: false, error: "..." }` and the agent decides what to do with it
- Multi-user support: planned (deferred; requires auth, session isolation, and remote deployment groundwork)
- Remote/cloud deployment: planned (deferred; local-only through v0.3)
- Non-developer users: not in scope
- Non-Claude Code agent runtimes: planned
- Slideshow / auto-play (`slideshow()`, `slideshow_stop()`): shipped in v0.2

## 7. Architecture Consolidation (v0.23–v0.26)

> Promoted from `desing-analysis/` (FR22 in `01`) via a `/grill-me` stress-test during intake, 2026-07-07. See `02` §N for the adoption-scope/sequencing decisions and `04` §9 for the full target architecture (unit map, decision points D1–D5, contract changes, bug traceability). These are internal-structure requirements, in the same spirit as NF12/NF13 (the v0.21 shared-core/dynamic-import refactors) — NF14–NF16 are output-equivalence requirements (no external behavior change), NF17 is the one deliberately-breaking requirement.

| ID | Requirement | Milestone |
|----|-------------|-----------|
| NF14 | **Unified WebSocket projector:** one shared broadcast-builder function replaces the 13 independently hand-built `{ action: "replace", ... }` construction sites across `app.ts`, `mcp.ts`, `render-core.ts`, `slideshow.ts`, and `ws.ts`. Every broadcast it produces carries `id` + cursor + viewport, unconditionally — this is the structural fix for the whole B15/C2d drift class (a later broadcast producer forgetting a field the others already have). Message shape (fields present) must not change — output-equivalence, verified by the full existing test suite passing unchanged. | v0.23 |
| NF15 | **Client renderer registry:** a `type → component` registry replaces `App.svelte`'s hardcoded `{#if canvas.type === "mermaid"} … {:else if …}` dispatch chain. Adding a future renderer type requires one registration, not a chain edit. Preserves the existing v0.21 per-type dynamic `import()` lazy-loading behavior. No behavior change — verified by the existing e2e suite passing unchanged. | v0.24 |
| NF16 | **Persistence policy + finalize dedup:** every command path (`render`, one-shot/incremental step-frames, slideshow) declares an explicit persist trigger — `immediate \| on-finalize \| transient \| never` — and routes through one shared finalize/persist implementation, replacing `slideshow.ts`'s `finalizeSlideshow()` re-assembly and the duplicate finalize logic in `render-core.ts`. A new command literally cannot omit a persistence decision — the direct structural fix for the FR20/B15 "silently never persisted" bug class. Existing persistence semantics (F7/F10/F15) must be preserved exactly — verified by the existing test suite passing unchanged. | v0.25 |
| NF17 | **Unified Presentation/Frame content model (breaking change):** `Presentation { id, title?, cursor, frames: Frame[] }` / `Frame { type, payload, label? }` replaces the three parallel content shapes (bare `render`, step-frames envelope, slideshow envelope) everywhere — `session.ts`'s and `canvasStore.ts`'s `CanvasState` unions, the WS contract (`stepFrames` boolean → `cursor`/`total` fields), the on-disk snapshot schema, and the MCP payload shape (`type:"step-frames"` removed as a top-level content type; `render`/`slideshow`/`step`/`seek` verbs are kept as ergonomic sugar per D5 in `04`). Requires: (a) a one-time deterministic snapshot-migration script, run only after a verified backup of `~/.agent-whiteboard/` (see `02` N5); (b) `README.md` and `tests/human_driven/showcase.js` updated to the new contract; (c) a full rewrite of every unit/e2e test asserting the old WS/payload/snapshot shapes — no back-compat shim, per `02` N4 (pre-public-release, zero external consumers). This milestone must complete before any public release. | v0.26 |
