# Milestone v0.30 — Export HTML Layout & Navigation Fixes (Sprints 66–68)

**Status:** released

> Opened 2026-07-15 via `/doc-creator-driver:intake` (bug report). User tested a real export file (`study-coach_algorithms-20260713-201859.html`) on a wide desktop viewport and found three layout defects plus one navigation gap in `server/export-html.ts`'s assembled document (F17 in `03`, see B20–B22 in `01`, `5d` in `03`). All four fixes are scoped to `export-html.ts`'s `LAYOUT_CSS` constant and `assembleHtml()` — no other module touched, no architecture change (see `04`'s HTML Export data flow, updated in place).

### Sprint 66 — Widen `<main>` and contain overflow (B20)
- [x] **B20.** Fix `LAYOUT_CSS` so `<main>` makes effective use of available viewport width on wide screens instead of rendering as a narrow fixed-feeling column, and so no contained element (table, `<pre>`/code block) ever visually exceeds its enclosing `.item-section`/`.frame-section` border — wrapping or scrolling internally instead (e.g. `overflow-x: auto` on table/code containers).
  - *First pass (overflow containment) — real, but not the reported bug's actual cause:* found via Playwright repro that an oversized table/`<pre>` with no `overflow-x` containment overflowed `.item-section`'s border and propagated scrollable overflow up to `<body>`, expanding the whole page's scroll width. Fixed via `overflow-x: auto` on `.item-section`/`.frame-section` plus `max-width: 100%` on `table`/`pre`/`code`. **Manual re-test after this landed showed the reported "~50% of screen width" narrowness unchanged** — this fix was real (worth keeping) but wasn't the cause of the user's actual complaint.
  - *Real root cause (found after manual-test gate reported no change, confirmed via Playwright repro):* an `html`-type payload is inserted into `.item-section` after only `DOMPurify` sanitization (`USE_PROFILES: { html: true }`), which does **not** strip `<style>` elements. A `<style>` tag is document-scoped, not scoped to the div it's nested inside — a content generator that ships its own theme stylesheet (e.g. a markdown→HTML converter's readable-width CSS, such as `body { max-width: 900px; margin: 0 auto; }`) silently overrides `LAYOUT_CSS`'s own `body`/`main` rules for the entire exported document, regardless of anything `LAYOUT_CSS` itself sets. Confirmed: reproduced the exact symptom (`body` clamped to the payload's `max-width`) with a synthetic payload.
  - *Second pass (`FORBID_TAGS: ["style"]`) — closed the leak but broke a different thing:* stripping `<style>` outright fixed the width, but the manual-test gate then reported the payload's own table/code formatting (dark header, row highlighting) had disappeared — the same `<style>` block the payload used to leak `body { max-width }` also carried its *legitimate* local formatting, and blanket-forbidding the tag threw both out.
  - *Third pass (`@scope`, real fix) — contains without discarding:* replaced `FORBID_TAGS` with a `scopeEmbeddedStyles(html, anchorId)` post-processing step in `assembleHtml()` that wraps any `<style>...</style>` found in rendered content in `@scope (#${anchorId}) { ... }`, where `anchorId` is that content's own `itemId`/`frameId`. `@scope` limits *selector matching* (not just specificity) to the scope root's subtree: a `body {}` rule inside the scope now matches nothing (no `<body>` descendant of `#item-N` exists), while `table {}`/`td {}`/etc. still match the payload's own elements normally. Verified support in this project's target runtime (Playwright's bundled Chromium). Re-verified with a payload carrying both an offending `body { max-width }` rule and legitimate table styling: the export now fills the viewport correctly **and** the table's own dark header/row-highlight styling renders intact.
  - *Acceptance:* opening the export on a wide (≥1600px) viewport shows `<main>` filling the space alongside the fixed-width `<nav>`; a table or code block wider than its section scrolls horizontally within that section instead of bleeding past its border.
  - *Regression coverage:* new test in `tests/unit/server/export-html.test.ts` asserting the relevant CSS rules are present in assembled output; manual visual check against the reported file.

### Sprint 67 — Fix table border alignment (B21)
- [x] **B21.** Fix the table CSS so borders render as a clean, uniformly aligned grid — every row's right edge flush with the table's outer border, no per-row drift (likely a `border-collapse`/`box-sizing` interaction in the current table styling).
  - *Root cause (found via Playwright repro):* no `table` rule set `border-collapse`, so tables used the browser default `border-collapse: separate` plus its default `border-spacing`, which visibly offsets a table's own outer border from its cells' borders (confirmed: table's right edge measured 3px right of every row's right edge). Fixed by adding `border-collapse: collapse` to the `table` rule in `LAYOUT_CSS` — confirmed via the same repro that the offset drops to sub-pixel (0.5px rendering rounding, not visible).
  - *Acceptance:* tables in the exported HTML show no visible misalignment between row borders and the table's outer border, matching `./unaligned_right_border.png`'s reported defect resolved.
  - *Regression coverage:* manual visual check against the reported file (table-border alignment is not practically assertable via DOM/CSS unit tests).

### Sprint 68 — Per-frame nav submenu for step-frames items (B22)
- [x] **B22.** In `assembleHtml()`'s TOC-building loop, for a `stepFrames` item emit one submenu `<li><a>` per frame (frame's own label, falling back to `Frame N`) nested under that item's TOC `<li>`, with anchors matching the existing `${itemId}-frame-${i}` ids already used by `<main>`'s frame sub-sections. The item's own (parent) TOC link points at the frame-0 anchor (`${itemId}-frame-0`) instead of `${itemId}`, so it behaves identically to the first submenu entry.
  - *Implemented:* the TOC-building branch now checks `content.kind === "stepFrames"` before emitting; step-frames items get `<ul class="toc-frames">` nested under their `<li>`, one entry per frame. Non-step-frames items are unaffected (single flat `<li>`). Verified visually via Playwright screenshot (nested, indented, smaller-font submenu) and via unit tests asserting exact anchors/labels for both the parent-to-frame-0 link and each submenu entry.
  - *Acceptance:* exporting a step-frames snapshot shows one submenu entry per frame in the left nav; clicking the parent item entry and clicking the first submenu entry scroll to the same place; clicking any other submenu entry jumps to that frame's section.
  - *Regression coverage:* new test in `tests/unit/server/export-html.test.ts` asserting the TOC HTML contains one `<a href="#...">` per frame for a step-frames item, and that the parent link's `href` matches the frame-0 anchor.
