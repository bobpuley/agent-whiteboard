### Sprint 35 — Showcase Coverage, Step-Frames Fit Fix & Slideshow Persistence
- [x] **fix: slideshow-driven renders never auto-fit (bug B15, see `01`)** — `server/slideshow.ts`'s `broadcastSlide()`/`broadcastTick()` never included an `id` field in their WebSocket broadcasts, so the browser's `isNewSnapshot()` check (F19) was always false for any `POST /slideshow`-driven content. Fixed by generating and threading a snapshot id through every slideshow broadcast — fresh per plain slide, one id shared across a step-frames sequence's frame ticks (continuation semantics, matching `/step`/`/seek`). Necessary but, as B17 below revealed, not sufficient on its own.
- [x] **feature: showcase feature-coverage audit (FR19, see `01`)** — audited Sections 1–12 against `03_requirements.md`'s full feature list (excluding delete/export per user instruction); found two shipped MCP features with no demo section: the incremental step-frames protocol (`init_step_frames`/`append_frame`/`commit_step_frames`, F15) and `node_to_frame` autonomous navigation (U4e). Added as Section 13 and Section 14 to `tests/human_driven/showcase.js`, with new `-c`/`--incremental` and `-n`/`--nodetoframe` flags (folded into `-a`/`--all`).
- [x] **fix: `node_to_frame` clicks 404 in dev mode (bug B16, see `01`)** — found while verifying Section 14: `client/vite.config.ts`'s dev proxy never listed `/seek`, the one endpoint the browser calls directly for `node_to_frame`. Added the proxy entry. Production (single-origin) builds were never affected.
- [x] **feature: slideshow finalize-on-end persistence + required `workspace` (FR20, see `01`)** — `POST /slideshow` never called `saveSnapshot()` at all (going back to Sprint 9), so nothing shown via a slideshow ever appeared in History. User proposed a "transient until finalized" model mirroring `init_step_frames`/`append_frame`/`commit_step_frames` (F15): individual ticks stay transient, exactly one snapshot is written when the session ends (natural completion, explicit stop, or supersession by a new `render()`/`slideshow()` call), capturing whatever was last on screen. Required adding a `workspace` parameter to `slideshow()`/`POST /slideshow` (previously had none, unlike `render()`'s F14 mandatory pattern) — user chose explicit-required over falling back to `lastWorkspace`. `clear()` deliberately does not finalize (`cancelSlideshow({ persist: false })`), preserving F10.
- [x] **fix: the real root cause — Mermaid SVGs have no explicit pixel size, so some browsers substitute the CSS default replaced-element size instead of the viewBox's actual size (bug B17, see `01`)** — even after B15's fix, the user still saw unfit diagrams *live* (though History-reloaded content was always correct). Console inspection showed `fitToView()`'s computed `scale` was mathematically correct against the `viewBox`, but `getBoundingClientRect()` proved the SVG was actually rendering at ~300×150 (the CSS spec's default replaced-element size) — because Mermaid emits `width="100%"` with no explicit height, and `Mermaid.svelte`'s CSS deliberately leaves the container unsized. Headless Chromium (used for all automated reproduction during B15's investigation) resolves the percentage differently and never hit this fallback, which is why every automated repro looked fine and only the user's real browser reproduced it. Fixed by explicitly pinning the SVG's `width`/`height` attributes to its `viewBox` dimensions immediately after insertion. Confirmed fixed live by the user.
- [ ] **not implemented — re-fit per frame for step-frames sequences of varying size (FR21, see `01`)** — fixing B17 exposed the residual risk already documented in C3 (`02`): a step-frames sequence's single shared fit (computed once at frame 0) can overflow/under-fill later frames of very different size (confirmed live in Sections 6, 7c, and 8). User chose "re-fit on every frame" as the intended direction but explicitly asked to intake the decision only. **Deliberately left undone** — unscheduled, no milestone assigned.

> **Implementation note:** B15 was diagnosed and implemented first — root cause required live reproduction to isolate from several ruled-out theories (window size, CSS timing, mermaid style directives) before screenshots from the user's own run pinned it to Section 6/`slideshow.ts`. B16 was an unplanned discovery during Section 14 verification. FR20 (persistence) emerged from the user's own suggestion of a calmer way to re-test B15, and needed its own sub-decision (workspace-source, then persistence granularity) before implementation. B17 was found only after the user re-tested B15's fix live and reported the bug "still there" — a second, deeper root cause that automated Playwright reproduction never surfaced (a genuine headless-vs-real-browser CSS layout difference), found via direct browser console inspection (`getComputedStyle`, `getBoundingClientRect`, `viewBox`) rather than further automated repro attempts.
